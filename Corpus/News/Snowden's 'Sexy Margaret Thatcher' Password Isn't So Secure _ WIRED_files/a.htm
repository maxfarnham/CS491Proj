                                ramriot a day ago

                                Even though this was a humerus exchange (at least on Edward's side) it has something to say about security and the perception of strength in passwords. The other commentators are correct that a 27 Character mixed case password with punctuation would take 8.04 hundred thousand trillion trillion centuries to brute force, assuming no smart algorithm, and yet is impossibly week should you know that the sexual allure of the ex PM is a password thing. Unfortunately what wired et-al fails to address a solution. Because by their and any reasonable definition a strong password is not a memorable password (especially if the service limits the length allowed, to keep the service call volume down), also that each service requiring a password MUST be given a different one.

                                So by way of some food for thought here are a few options the reader may try:-

                                a) Use a password safe (Lastpass, KeyPass etc), have a single strong passphrase to unlock it. Generate maximum length random Alphanumeric and punctuation passwords for each site and use the password safe to remember them.

                                b) Use a strong password generating algorithm that can be run offline ( https://www.grc.com/OffTheGrid... ) and used with the sites domain as input to generate strong deterministic passwords. Optionally store these in a password safe as above for convenience.

                                c) Use multi-factor authentication as a backup to a less than totally strongpassword.

                                d) Push your most important services to offer an alternative authentication system that does not need any secret information like a password to be stored by the service but relies on psudononymous site-specific zero knowledge proofs. i.e. in the near future SQRL ( http://sqrl.pl/guide/ ) and FIDO ( https://fidoalliance.org/ )

                                                                                                                                  Well the whole idea of a single layer password is inherently a bad foundation. P(defense A or defense B) = P(defense A)+P(defense B) - P(defense A and defense B). Which is just saying a defense working with a password and a text key code is much higher than just a text key code. They can have the password, but then actually fail at still getting in because of the second layer. But this is still based on a static password. If one could count the number of ways to provide security access between entity A and entity B, I'm sure it would be past trillion member set. Passwords are just one way. But the Multi-Layer perspective gives a framework at choosing a structural foundation for good choices. So for example the FBI confiscates a computer to hack. The whole framework is that one enters a password for the password or encryption. And they might have got that by some crafty means. But if it has mutli layers, and something unfamiliar to them it becomes vastly more less likely for success. And thats the whole deal with P(A or B) = P(A)+P(B)-P(A&B).. and when P(A&B) falls to near 0 or 0, then the chances of successful defense are being added. I think Snowden would do better to redefine the password. If he remembers an array of numbers, then the server could flash him a pattern, then from his password array the password is based on that pattern. Now if someone somehow got the password, its only good for that 1 entry or till that pattern shows up again. Next time he goes to log in, there another pattern that flashes. So the password entered in is not the actual password, but based on the array and pattern.. giving it a dynamic changing ability which would require a much longer period of observation to get past. So even if they knew 4 or 5 pattern keys, Snowden would still have a pretty good chance of his account being safe. Which would be good for admin levels, where people have strong ability to use alternative access checking methods.
